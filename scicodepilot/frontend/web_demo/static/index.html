Memory-Augmented Structured Agent

Reliable Scientific Code Diagnosis and Safe Repair

SciCodePilot studies whether structured failure memory, retrieval, constrained patch planning, safety review, and isolated verification can make scientific Python repair more auditable and reproducible than direct traceback-to-patch generation.

Research Problem

Why not directly ask an LLM to fix the traceback?

Scientific Python and machine learning experiments often fail because of tensor shape, dtype, device, loss input, collate function, config key, missing dependency, or missing data-file issues. These failures are usually local, but they strongly affect reproducibility.

Direct LLM patching tends to produce free-form edits from the current traceback alone. SciCodePilot instead turns the failure into an inspectable intermediate state, routes environment and data issues away from source patches, and records evidence before and after repair.

RQ1 Structured diagnosis

Can raw stderr and traceback be converted into stable FailureMemory?

RQ2 Safety-aware routing

Can the system avoid patching source code for dependency/data failures?

RQ3 Constrained planning

Are structured PatchPlans easier to review than free-form patches?

RQ4 Memory augmentation

Can retrieved FailureMemory examples support future repair planning?

Method

A structured repair pipeline with explicit safety boundaries

1Run task and capture stdout/stderr
2Parse traceback into FailureMemory
3Route source, environment, and data issues
4Generate PatchPlan or EnvRepairPlan
5Review, confirm, apply, and verify

FailureMemory Retrieval

Failures are stored as reusable, searchable agent memory

FailureMemory is the central intermediate representation in SciCodePilot. It stores the failure type, evidence, root-cause hypothesis, repair action, command context, and verification result so future failures can retrieve similar cases.

The current retrieval evaluation is deterministic and scoped to the internal benchmark. It demonstrates that the memory layer can recover matching cases and build in-context examples, not that the system has completed large-scale semantic retrieval.

Safety and Ablation

Repair is gated by review, confirmation, and isolated verification

PatchSafetyReviewer checks risky patterns such as path traversal, dependency installation, shell-command edits, benchmark mutation, and overly broad multi-file changes before any patch is applied.

The ablation assets separate diagnosis-only, repair without apply, full rule-based repair, mock LLM repair, and safety stress cases so the report can explain which component contributes to reliability.

Risk class Expected system behavior
Path traversal Block patches that attempt to write outside the allowed workspace.
Dependency installation Route to EnvRepairPlan instead of silently running package installs.
Missing data Report data/path issue instead of fabricating files.
Benchmark mutation Reject changes that alter expected benchmark criteria.

Positioning

Architectural comparison without claiming public benchmark superiority

SciCodePilot is positioned as a controlled, auditable repair prototype. It should not be described as outperforming SWE-agent, OpenHands, AutoCodeRover, or mini-swe-agent on public benchmarks. The comparison focuses on design dimensions: memory, routing, structured planning, safety review, and reproducibility evidence.

          External Repository Interface

          A smoke entry point beyond built-in benchmark tasks

          python scripts/run_external_repo_smoke.py --repo-path <local_repo> --command "pytest -q" --mode diagnosis

          The external smoke interface copies a local repository into an isolated workspace, runs the provided command, and writes summary.md and summary.json. It does not mutate the original repository and should not be reported as public benchmark performance.
